## 0. 摘要

M5 完成两件事:

1. **节点批量更新 UX**:用户从单节点逐个点击 update,改为跨 env 全选一次性 `git pull`。痛点:10 env × 20 节点 = 200 次手动点击。
2. **M4 carry-over 收尾**:M4 review ledger 留下的 5 项技术债合并 / 修复。

实现形态:

- **Backend**:1 个新 service (`BulkUpdateService`)+ 1 个新 routes 文件 + 3 个新 endpoints
- **WPF**:1 个新 Dialog (`BulkUpdateDialog`)+ 1 个新 ViewModel + 1 个 WS 订阅
- **Cleanup**:5 个 carry-over task(repo shim 合并、4 个 pre-existing 测试修)

**REST + WS delta**:3 endpoints 新增、1 WS channel 新增、2 pydantic model 新增。

## 1. 目标 & 非目标

### 1.1 目标(M5 完成时)

- 用户在 WPF 打开 `BulkUpdateDialog`,可视化 10 env × N 节点的网格
- 顶部 select-all checkbox 一次勾选全部
- 用户点 Start → 后台串行执行 `node.update`,每行完成有进度推送
- 失败 / 跳过 / 取消均有清晰 UI 反馈,关闭 dialog 后历史 record 仍可查
- Carry-over 5 项全清(架构干净 + tests 全绿)

### 1.2 非目标(明确不做)

- ❌ git fetch 预检("哪个节点有可用更新"badge)—— 简化,v0 只盲目 git pull(幂等)
- ❌ Per-env 并发 —— v0 全串行,git lock 不会冲,简单优先
- ❌ 失败节点重试 —— 用户看到失败的,自己手动处理
- ❌ Persistent history(BulkUpdateSummary 落库)—— v0 only in-memory
- ❌ 跨 catalog 跨源(只跨同 repo 的 env 间一致)
- ❌ 自动提示 / 后台定时 bulk —— v0 全按需触发

## 2. 架构

### 2.1 总体布局

新增 **BulkUpdateService**(Python 层)负责跨 env 调度 `node.update`,通过 M4 已有的 `WSBroadcaster` 推送 per-row 进度。**BulkUpdateDialog**(WPF)订阅 `/ws/events` channel `bulk_update`,用 DataGrid 展示行级结果。**Cancel token** 通过新 REST endpoint `POST /bulk-update/{id}/cancel` 传回。

### 2.2 数据流

```
User opens BulkUpdateDialog
  ↓ fetch env list + node list per env (existing endpoints, no new schema)
User picks {envs × nodes} → click Start
  ↓ POST /api/v1/bulk-update/start  {env_ids, node_ids}
Service 立即返回 {bulk_id}
  ↓ WS push bulk_update.started → dialog 进入 running 态
Service 后台串行(每 env 顺序,env 内顺序)调 node.update(已有 service,无改)
  ↓ 每完成 1 行 push bulk_update.progress {env_id, node_id, status}
User 点 Cancel → POST /api/v1/bulk-update/{id}/cancel
  ↓ service 检查 token,在下个 (env, node) 边界停(已完成 row 不回滚)
全部完成 → push bulk_update.completed {summary: succeeded / skipped / failed}
dialog 切到 summary view(succeeded=N, skipped=M, failed=K + 失败原因列表)
```

**Dialog 关闭行为**:用户关闭 dialog 不取消 service 端 bulk;service 继续跑。下次打开 dialog 调 `GET /bulk-update/{id}`,若有 in-flight → 显示 running 视图;若已完成 → 显示 summary。Service 重启后 history 丢失(404),dialog 显示 "previous run no longer available"。

### 2.3 关键设计决定

| 决定 | 选择 | 理由 |
|---|---|---|
| Git 预检 | 不做 | 简化;git pull 幂等,already-up-to-date 秒退 |
| 失败策略 | best-effort + summary | 失败节点不阻断后续;interrupt UX 反而是阻力 |
| Cancel 粒度 | (env, node) 边界 | 半行 cancel git 操作易锁;选干净断点 |
| 并发 | 全串行 | git index.lock 不会冲;v0 不优化 |
| 结果持久化 | 内存 only(per service lifetime) | "small feature",dialog 关了不存 |
| Bulk id 唯一性 | UUID4 | 进程内唯一足够 |

## 3. 接口契约

### 3.1 REST endpoints(3 个)

#### `POST /api/v1/bulk-update/start`

**Request**:
```json
{
  "env_ids": ["env-1", "env-2"],
  "node_ids": ["node-a", "node-b", "node-c"]
}
```

**Response 200**:
```json
{ "ok": true, "value": { "bulk_id": "uuid-xxx" } }
```

**Response 4xx**:
- `400 VALIDATION` — env_ids 或 node_ids 为空
- `404 NOT_FOUND` — env_id 不存在
- `409 ALREADY_RUNNING` — 同 bulk_id 已 active(防止重复)

#### `POST /api/v1/bulk-update/{bulk_id}/cancel`

**Response 200**:
```json
{ "ok": true, "value": { "cancelled_at_checkpoint": "env-2#node-b" } }
```

**Response 4xx**:
- `404 NOT_FOUND` — bulk_id 不存在
- `409 NOT_RUNNING` — 已 completed / cancelled,无 effect

#### `GET /api/v1/bulk-update/{bulk_id}`

**Response 200**:
```json
{
  "ok": true,
  "value": {
    "bulk_id": "uuid-xxx",
    "status": "running",   // pending | running | completed | cancelled | failed
    "started_at": "2026-07-11T08:00:00",
    "finished_at": null,
    "total": 6,
    "succeeded": 2,
    "skipped": 0,
    "failed": 0,
    "current": "env-1#node-c"
  }
}
```

启动 dialog 时用 `GET` 拉历史 summary(若 service 仍记着)。

### 3.2 WebSocket event channel(1 个)

**Channel**: `bulk_update`

| Kind | Payload |
|---|---|
| `bulk_update.started` | `{ bulk_id, total, env_ids, node_ids }` |
| `bulk_update.progress` | `{ bulk_id, env_id, node_id, status: "succeeded"\|"skipped"\|"failed", reason?: str, latency_ms: int }` |
| `bulk_update.completed` | `{ bulk_id, summary: BulkUpdateSummary }` |
| `bulk_update.cancelled` | `{ bulk_id, summary: BulkUpdateSummary, cancelled_at_checkpoint: str }` |
| `bulk_update.failed` | `{ bulk_id, error: { code, message } }` |

**`BulkUpdateSummary` payload**:
```json
{
  "total": 6,
  "succeeded": 5,
  "skipped": 1,
  "failed": 0,
  "rows": [
    {"env_id": "env-1", "node_id": "node-a", "status": "succeeded", "latency_ms": 1200},
    {"env_id": "env-1", "node_id": "node-b", "status": "skipped", "reason": "uncommitted changes"}
  ]
}
```

### 3.3 Pydantic schemas(M4 schemas.py 末尾追加)

```python
class BulkUpdateRequest(BaseModel):
    env_ids: list[str] = Field(min_length=1)
    node_ids: list[str] = Field(min_length=1)

class BulkUpdateStartedResponse(BaseModel):
    bulk_id: str

class BulkUpdateRow(BaseModel):
    env_id: str
    node_id: str
    status: Literal["succeeded", "skipped", "failed"]
    reason: Optional[str] = None
    latency_ms: int = 0

class BulkUpdateSummary(BaseModel):
    total: int
    succeeded: int
    skipped: int
    failed: int
    rows: list[BulkUpdateRow]

class BulkUpdateStatus(BaseModel):
    bulk_id: str
    status: Literal["pending", "running", "completed", "cancelled", "failed"]
    started_at: Optional[str] = None
    finished_at: Optional[str] = None
    total: int
    succeeded: int
    skipped: int
    failed: int
    current: Optional[str] = None  # "env-x#node-y"
```

## 7. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| WPF DataGrid 在 200+ 行时性能下降 | dialog 卡顿 | 用 `VirtualizingStackPanel.IsVirtualizing="True"` + `EnableRowVirtualization="True"`(DataGrid 默认开) |
| Service 后台跑 bulk 时 main UI 卡 | 用户体验差 | 严格在 service 后台(进程隔离),UI 只订阅 WS |
| Cancel 时 (env, node) 边界处理不对 | 部分 row 状态错 | 单测覆盖;在 git 操作前检查 cancel token |
| Carry-over CO-1 改 shim 后 import 链漏修 | service 启动失败 | 全项目 `grep "from comfy_mgr.app_context"` 一次性补齐;加 `tests/test_imports.py` sanity check |
| Smoke `/healthz` 根因非路由问题 | CO-5 卡住 | 调查完若是别的问题,fallback 写"已知 issue / health via /api/v1/health" |
